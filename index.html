ThreadPoolExecutor

ThreadPoolExecutor是ExecutorService的一个实现类，是java中最常见的线程池类。

构造函数

以参数最多的构造函数为例，分析各个参数的作用。

参数 参数格式 参数解释
corePoolSize int 线程池维护线程的最小数量
maximumPoolSize int 线程池维护线程的最大数量
keepAliveTime long 线程池中空闲线程的空闲时间，超过规定时间会回收
unit TimeUnit 空闲时间的单位
workQueue BlockingQueue 线程池所使用的缓冲队列
threadFactory ThreadFactory 设置的线程工厂类
handler RejectedExecutionHandler 线程池对拒绝任务的处理策略

corePoolSize与maximumPoolSize

当一个线程被添加到线程池后，具体的分配规则如下:

  • 如果此时线程池的数量小于corePoolSize，则每来一个任务，都会创建一个新的线程执行这个任务。
  • 如果此时线程池的数量大于等于corePoolSize，则每来一个任务，会尝试添加到任务缓冲队列当中，若添加成功，则该任务会等待空闲线程来执行此任务。如果添加
    失败，表示任务缓冲队列已满，则会创建一个新的线程来执行这个任务。
  • 如果此时线程池的数量大于corePoolSize，如果某个线程的空闲时间超过keepAliveTime，则线程将会被终止，直至线程池的线程数目等于corePoolSize。
    即使规定核心线程的存活时间，线程也会被终止。
  • 如果此时线程池的数量大于或等于corePoolSize，则会采取拒绝任务策略来处理。

workQueue

workQueue是一个BlockingQueue，默认是LinkedBlockingQueue。

当我们创建线程池的时候，如果没有新任务进来的话，默认是没有线程的。提交任务后才会创建新的线程。如果想要初始化corePoolSize数量的线程的话，线程池提供了
以下两个方法:

  • prestartCoreThread() 立即初始化一个线程
  • prestartAllCoreThreads() 立即初始化corePoolSize数量的线程

当创建的线程数量大于corePoolSize后，新来的任务会被加入到堵塞队列中等待有空闲线程来执行。类型为BlockingQueue。有以下三种实现类型

  1. ArrayBlockingQueue

基于数组的先进先出队列，是有界的，创建时必须制定大小。

可以防止资源耗尽。但是在java中不推荐使用。

  1. LinkedBlockingQueue

基于链表的先进先出队列，是无界的，默认大小是Integer.MAX_VALUE。

默认创建线程池的时候，策略就是这个。但是也可以制定大小使之变成有界的。当为无界时，在线程数量等于corePoolSize之后，永远不会创建新的线程。
所有增加的线程都会被添加到缓存队列中。这样的话资源会有被耗尽的危险。

在Executors中的实现方式

2.1 newFixedThreadPool

可以看到，创建了制定线程个数的线程池，当线程空闲时候，立马进行回收(但是由于线程永远都是这些，所以并不会回收，线程还是在的)。制定线程数量当线程池使用的也是无界队列。

2.2 newSingleThreadExecutor

可以看到，创建一个有唯一线程的线程池。所有的任务都按照顺序执行。单个线程池使用的也是无界队列。

  1. SynchronousQueue

无界队列，不会保存任务，每一个新增任务的线程必须等待另一个线程取出任务，这个队列只能添加一个任务，并且这个任务只有被执行完成之后，
才能继续添加。否则就要创建线程来执行任务，所以使用SynchronousQueue最好线程池最大数量为Integer.MAX_VALUE。

3.1 newCachedThreadPool

可以看到，创建一个最大数量的线程池。当线程空闲时间为60s之后，使之回收。

handler

拒绝任务的处理策略。

  • ThreadPoolExecutor.AbortPolicy()。系统默认，抛出java.util.concurrent.RejectedExecutionException异常。
  • ThreadPoolExecutor.CallerRunsPolicy()。当抛出RejectedExecutionException异常时，会调用rejectedExecution方法。
  • ThreadPoolExecutor.DiscardOldestPolicy()。抛弃旧的任务。
  • ThreadPoolExecutor.DiscardPolicy()。抛弃当前的任务。

ExecutorService

ExecutorService是对线程池定义的接口。接口定义了后台任务执行的相关方法。

方法说明

shutdown()

线程池使用完之后，应该进行关闭。调用shutdown方法，ExecutorService不会马上关闭，但是不再接收新的任务，直到当前线程执行完成之后才会关闭。
所以shutdown之前的提交的任务都会被执行。

shutdownNow()

执行这个方法的时候，会跳过正在执行的任务和被提交还没有执行的任务，然后依次执行线程的interrupt中断方法。但是只会中断阻塞的方法，对正在执行的任务，
并不能保证是否能停止。并且会返回未执行的任务。

isShutdown()

当调用shutdown或shutdownNow方法后返回为true。

isTerminated()

当调用shutdown后，并且所有提交的任务都完成后返回true。
当调用shutdownNow后，成功停止后返回true。

awaitTermination(long timeout, TimeUnit unit)

等待设置的时间，来判断当前任务是否执行完。一般与shutdown搭配使用来判断是否任务执行完成。

execute(Runnable command)

异步执行任务，但是没有办法获取任务执行的结果。

submit(Callable task)

异步执行任务，返回Future对象，并且接收的是callable，有返回值，可以通过future.get获取。future.get方法会阻塞线程。

结果为:

submit(Runnable task, T result)

返回Future对象。但是接收的是Runnable对象，是没有返回值的，第二个参数就规定了一个返回值，然后通过future.get获取的就是第二个参数。

结果为:

submit(Runnable task)

返回Future对象。但是接收的是Runnable对象，是没有返回值的。通过future.get获取的是null。

结果为:

invokeAll(Collection<? extends Callable> tasks)

多个任务一起执行，返回所有任务执行完成的集合，返回结果是没有先后顺序的。

结果为:

invokeAll(Collection<? extends Callable> tasks,long timeout, TimeUnit unit)

结果为:

同时执行所有任务，限制超时时间。在规定时间内没有完成，会报CancellationException异常，表示此方法超时。注意创建线程池的方式，如果是多个线程，
则会统一分配。如果只是一个线程，那所有任务都是一个线程执行的。一个线程超时的话，那所有任务都显示超时。

invokeAny(Collection<? extends Callable> tasks)

多个任务一起执行，返回运行最快的值。只返回一个值。

结果为:

invokeAny(Collection<? extends Callable> tasks,long timeout, TimeUnit unit)

在规定的超时时间内，获取返回值，如果没有，则表示超时。

构造方法

ExecutorService executorService = Executors.newFixedThreadPool(5)

创建有规定个数线程的线程池，超出的线程会在队列中等待。返回的是ThreadPoolExecutor。

ExecutorService executorService = Executors.newSingleThreadExecutor()

创建唯一线程的线程，保证所有任务都按照指定顺序执行。返回的是ThreadPoolExecutor。

创建一个可以缓存的线程池，如果线程池不够则主动扩充线程的个数。返回的是ThreadPoolExecutor。

ExecutorService executorService = Executors.newScheduledThreadPool(5)

创建规定个数线程的线程池，支持定时以及周期性任务执行。返回的是ScheduledThreadPoolExecutor。

ExecutorService executorService = Executors.newSingleThreadScheduledExecutor()

创建唯一线程的线程，支持定时以及周期性任务执行。返回的是ScheduledThreadPoolExecutor。

Thread

线程，可以看作是进程的一个实体，是CPU调度和分派的基本单位，是比进程更小的能独立运行的基本单位。

主要函数

run()

线程运行所执行的代码。调用的是runnable内的run方法，发生的线程是调用run方法的线程

start()

启动线程。启动线程来执行，发生的线程是创建的子线程

sleep(long millis),sleep(long millis,int nanos)

第一个方法表示休眠多少毫秒，第二个方法表示休眠多少纳秒，为第一个参数毫秒转成纳秒(*1000000)加上第二个参数。

线程休眠，交出CPU，让CPU执行其他任务。但是sleep方法不会释放锁，也就是说如果当前线程持有某个对象的锁，即使调用sleep方法，其他线程也无法访问这个对象。

控制台输出为

从上面的例子可以看出，当Thread-0进入休眠之后，Thread-1并没有马上执行任务，当Thread-0任务执行完成之后，Thread-1才开始执行。

如果调用了sleep方法，必须捕获相关异常。当睡眠时间完成之后，不一定会立即得到执行，因为此时可能CPU正在执行其他任务，所以说sleep相当于让线程进入阻塞状态。

yield()

调用yield方法会让当前线程交出CPU，让CPU去执行其他的线程。跟sleep类似，同样不会释放锁，但是yield没办法控制交出CPU的时间。
并且，yield只能让拥有相同优先级的线程有获取CPU执行时间的机会。

调用yield不会让线程进入阻塞状态，而是让线程重新回到就绪状态，只需要等待重新获取CPU执行时间。

join(),join(long millis),join(long millis, int nanos)

调用join方法会让当前线程执行完成之后，才会继续向下执行。调用无参的方法，则表示等到当前线程执行完成之后向下执行，调用有参的方法表示等待多长时间，如果在
等待的时间内完成，则马上向下执行，如果没有完成，则等待到设置的时间后，马上向下执行。join方法内部调用的是Object的wait方法，是会阻塞线程的。由于wait方法
会让线程进入阻塞状态，并且会释放线程占有的锁，交出CPU的执行权限，所以join方法与wait方法一样，释放线程锁，线程进入阻塞状态。

wait(),wait(long millis),wait(long millis, int nanos)

线程等待的方法，会让线程进入阻塞状态，并且会释放线程占有的锁，交出CPU的执行权限。

interrupt()

中断线程，只能中断阻塞状态的线程(join,sleep,wait等)。如果想要中断非阻塞状态的线程，则需要用可见性的volatile来标示boolean变量，来实现方法的不执行。

getId()

获取线程的ID，是静态变量，随着创建线程的增加而变大。

getName(),setName()

获取或设置线程的名字

getPriority(),setPriority()

获取或设置线程的优先级，区间为1-10，默认值为5

setDaemon(),isDaemon()

设置或判断当前线程是否是守护线程。守护线程依赖与创建它的线程，而用户线程不依赖。在主线程创建一个守护线程，当主线程执行完之后，守护线程也会随着消失，而用户线程则不会。

currentThread()

静态函数获取当前线程

isAlive()

判断当前线程是否是死亡状态

状态

  • 新建状态(new):实例化之后进入该状态
  • 就绪状态(runnable):线程调用start()之后，进入就绪状态等待CPU执行。这时只是表示可以运行，但不表示已经运行
  • 运行状态(running):线程获得CPU的执行，开始执行run方法内的代码
  • 阻塞状态(blocked):线程进入阻塞状态，比如:join,sleep,wait。等待触发条件，等待由别的线程占用的锁
  • 死亡状态(dead):线程运行完毕或者异常退出，可使用isAlive获取状态

ThreadLocal

线程本地存储

定义

ThreadLocal为线程本地存储，每个线程可以访问自己的内部变量。每个线程都可以通过set和get方法对这个局部变量进行操作，但不会和其他线程对局部变量进行冲突，实现了线程对数据隔离。

当我们需要在各个线程之间操作相同的对象，但每个线程内的对象都不会影响别的线程的对象的时候，可以使用ThreadLocal。

原理

  • 每个Thread维护着一个ThreadLocalMap的引用
  • ThreadLocalMap是ThreadLocal的内部类，用Entry来进行存储
  • 调用ThreadLocal的set方法时，实际上就是往ThreadLocalMap设置值，key是ThreadLocal对象，值是传递进来的对象
  • 调用ThreadLocal的get方法时，实际上就是往ThreadLocalMap获取值，key是ThreadLocal对象
  • ThreadLocal本身并不存储，只是作为一个key来从ThreadLocalMap获取value

内存泄露

ThreadLocal的生命周期跟Thread一样长，如果没有手动删除对应key就会内存泄露，而不是因为弱引用，想要避免内存泄露就实现remove方法。

ThreadLocal的目的是为了在线程中有属于自己的变量，并不是为了解决并发或者共享变量的问题。

原子性&&分子性&&有序性

java中线程通信主要使用共享内存的方式

原子性

原子性是指操作是不可分的。其表现在于对于共享变量的某些操作，应该是不可分的。例如：a++。对于共享变量的a的操作，实际上会执行三个步骤：

  • 读取变量a的值
  • a的值+1
  • 将值赋予变量a
    这三个步骤中的任何一个操作过程中，a的值被人修改，都会出现我们不希望的结果，所以我们必须保证是原子性的。java中锁的机制解决了原子性的问题。

可见性

可见性表示一个线程中对共享变量对修改，对另一个线程是可的。
在java内存模型中，如果一个操作执行的结果需要对另一个操作可见，那么这两个操作必然存在appens-before关系。

  1. 程序次序规则：一个线程内，按照代码顺序，书写在前面的操作先行发生于书写在后面的操作；
  2. 锁定规则：一个unLock操作先行发生于后面对同一个锁lock操作；
  3. volatile变量规则：对一个变量的写操作先行发生于后面对这个变量的读操作；
  4. 传递规则：如果操作A先行发生于操作B，而操作B又先行发生于操作C，则可以得出操作A先行发生于操作C；
  5. 线程启动规则：Thread对象的start()方法先行发生于此线程的每个一个动作；
  6. 线程中断规则：对线程interrupt()方法的调用先行发生于被中断线程的代码检测到中断事件的发生；
  7. 线程终结规则：线程中所有的操作都先行发生于线程的终止检测，我们可以通过Thread.join()方法结束、Thread.isAlive()的返回值手段检测到线程已经终止执行；
  8. 对象终结规则：一个对象的初始化完成先行发生于他的finalize()方法的开始.

所以，只要实现volatile关键字或者使用锁对机制，就能实现内存对可见性。

有序性

有序性是指程序在执行对时候，程序对代码执行顺序与语句对顺序是一致的。为什么会出现不一致的情况呢？是由于重排序的缘故。在java内存模型中，允许编译器和处理器对指令进行重排序，但是重排序不会影响到单线程程序的执行，却会影响到多线程并发程序的执行。比如
